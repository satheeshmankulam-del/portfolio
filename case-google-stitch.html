Google Stitch —
A Small Menu, A Big Difference

How a single overlooked context menu in Google's Stitch creates cascading friction — and how one small design change could fix the entire experience.

Product
Google Stitch
Type
Friction Audit
Focus
Context menu UX
Published
Medium · Design Bootcamp
01 — Problem

The menu you can't find
when you need it

Google Stitch is a design tool — users are actively creating and editing. At the moment of highest intent (just selected an element, ready to act), the most-needed actions are hidden behind an overflow menu buried in a corner of the toolbar.

"A design tool that makes designers work to find basic design actions is a product that misunderstands its own user."

The problem isn't that the actions don't exist — they do. The problem is that their location creates unnecessary context-switching at the exact moment when a user should be in flow.

02 — Research

How a small menu
breaks big flows

🎯
Actions happen in context
When a user selects an element, their mental model immediately expects contextual actions — not a journey to a toolbar. Every pixel of distance between the selected object and its available actions is friction.
🧠
Cognitive load of search
Having to remember where an action lives (toolbar? overflow? right-click?) pulls the user out of creative flow and into navigation mode. For a design tool, this is particularly costly — the work is inherently high-cognitive-load already.
📐
Established design tool patterns
Figma, Canva, and Adobe XD all surface contextual toolbars on selection. The mental model is already established. Deviating from it without good reason creates unnecessary relearning.
⚡
Small fix, large impact
A floating contextual toolbar that appears on element selection — with the 4-5 most common actions — eliminates the majority of the navigation friction without any structural redesign of the product.
03 — Design Process

Surface the actions
to the action

01
Identify the most-used actions per element type
Audited what users most commonly want to do after selecting an element — text: edit, style, duplicate, delete. Image: resize, replace, layer order. These top-5 actions per type form the contextual toolbar.
02
Design a floating contextual toolbar
A compact toolbar that appears proximate to the selected element on selection — not in a fixed toolbar position. Disappears on deselect. Adapts its action set based on element type.
03
Keep the overflow for advanced actions
The existing overflow menu isn't wrong — it just shouldn't be the only access point. Advanced or rarely-used actions stay in the overflow. Common actions move to the contextual toolbar.
04
Progressive disclosure in the toolbar
Primary actions are visible. Secondary actions appear on a "more" tap within the floating toolbar — maintaining compactness without hiding critical functionality.
04 — Outcomes

The principle of
contextual proximity

1
Toolbar interaction
to access any action
0
Navigation to toolbar
for common actions
∞
Design flows
that stay unbroken

Conceptual redesign published in Design Bootcamp. The core principle: actions should live where the intent lives. When a user selects something, they're ready to act — the design should meet them there.
